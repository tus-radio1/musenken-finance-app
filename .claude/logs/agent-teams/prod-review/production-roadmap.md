# 本番環境移行に向けた改善ロードマップ

本番環境の構築・移行に向けて、インフラ、コード品質、セキュリティの3つの観点で行われたレビュー結果（`infra-reviewer.md`, `quality-reviewer.md`, `security-reviewer.md`）を統合し、対応すべき課題と改善のロードマップをまとめました。

## 🚨 重大な課題 (Critical & High) - 最優先で対応が必要

レビューで発見された最も致命的な課題は以下の通りです。これらは本番稼働前に必ず修正する必要があります。

1. **認証・認可の欠如と脆弱性**
   - **ミドルウェアの欠如:** Next.jsのルート保護やSupabaseのセッション更新を行う `middleware.ts` が存在しません。（Critical）
   - **脆弱な初期パスワード:** 初期パスワードが学籍番号から決定論的に生成されており（例: `{yy}rc{student_number}`）、推測が容易です。また、初回ログイン時のパスワード変更の強制もありません。（Critical）
   - **認証なしのファイルアップロード:** `app/actions.ts` の `uploadReceiptAction` で認証チェックが行われておらず、管理者クライアント権限で誰でもファイルをアップロード可能な状態です。（Critical）
   - **管理者クライアントの乱用:** アプリケーション全体で `createAdminClient()` が頻繁に使用されており、Row Level Security (RLS) をバイパスしてしまっています。（Critical）
   - **RLSポリシーの欠如:** `fiscal_years` テーブル以外のすべてのテーブル（`transactions`, `subsidy_items`, `profiles` 等）にRLSポリシーが設定されていません。（Critical）

2. **データ整合性とアーキテクチャ**
   - **論理削除の未実装（物理削除の横行）:** 財務記録（`transactions`, `subsidy_items`）やユーザープロファイルが物理削除（ハードデリート）されており、監査要件を満たせていません。（Critical）
   - **監査ログの欠如:** 財務データの作成・更新・削除を追跡する `audit_log` テーブルが存在しません。（Critical）
   - **ハードコードされたUUID:** 複数のファイルで、特定のUUID（`9701edd2-bd9d-4d57-9dd6-7235686103bf`）が「会計システムユーザー」としてハードコードされており、環境間の移植性がありません。（High）
   - **個人情報のログ出力:** 本番コード（ログインページなど）に、ユーザーの認証データを出力する `console.log` が残っています。（High）

3. **エラーハンドリングと品質**
   - **エラーバウンダリの欠如:** アプリケーション内に `error.tsx` が存在せず、エラー時にアプリケーション全体がクラッシュまたは白画面になる可能性があります。（High）
   - **未認証時の挙動:** 認証されていないユーザーが特定のページ（`applications`, `subsidies`など）にアクセスした際、ログイン画面へのリダイレクトではなく、白画面やnullが返却されます。（High）

---

## 🗺️ 改善ロードマップ

これらの課題を計画的に解決するため、4つのフェーズに分けたロードマップを提案します。

### フェーズ 1: 致命的なセキュリティと認証の修正（即時対応）
本番環境にデプロイする前に、システムの不正利用を防ぐための最小限のセキュリティ要件を満たします。

- [ ] **ミドルウェアの導入:** `middleware.ts` を作成し、全ルートの保護とSupabase SSRセッションの自動更新を実装する。
- [ ] **パスワードポリシーの改善:**
  - 初期パスワードの生成ロジックをランダムで安全なものに変更する。
  - 初回ログイン時にユーザーにパスワードの変更を強制するフローを追加する。
  - サーバーサイドでのパスワード強度チェック（長さ、複雑さ）を実装する。
- [ ] **ファイルアップロードの安全化:** `uploadReceiptAction` に認証チェックを追加し、ファイルの種類とサイズのバリデーションをサーバーサイドで実装する。また、ディレクトリトラバーサル対策としてファイル名のサニタイズを行う。
- [ ] **機密情報のログ削除:** 本番環境のコードから `console.log`（特にログイン処理周り）を完全に削除する。
- [ ] **未認証時のリダイレクト:** 認証が必要なページへの未認証アクセス時に、ログインページへ適切にリダイレクトさせる処理を実装する。

### フェーズ 2: データ整合性と認可機構の構築（短期）
データの意図しない変更や漏洩を防ぎ、財務システムとしての信頼性を確保します。

- [ ] **RLS（Row Level Security）の全面適用:**
  - すべてのテーブルに対して、適切なRLSポリシーを設計し、マイグレーションファイルとして適用する。
  - データの取得・更新処理を `createAdminClient()` から通常のユーザー権限クライアント（`createClient()`）に置き換える。
- [ ] **論理削除の導入:** 財務データおよびユーザーデータの削除処理を、物理削除（DELETE）から論理削除（`deleted_at` カラムの更新など）へ移行する。
- [ ] **監査ログ（Audit Log）の実装:** データ変更履歴を記録するテーブルを作成し、トランザクションや金額変更の履歴を追跡可能にする。
- [ ] **ハードコードの排除:** ハードコードされている「会計システムユーザー」のUUIDを環境変数やデータベースの設定テーブルから動的に取得する仕組みに変更する。

### フェーズ 3: エラーハンドリングとコード品質の向上（中期）
システムの安定性を高め、保守しやすいコードベースを確立します。

- [ ] **エラーバウンダリの設置:** `app/` ディレクトリの適切な階層に `error.tsx` を配置し、システムエラー時にユーザーフレンドリーな画面を表示する。
- [ ] **サーバーアクションのバリデーション強化:** すべてのサーバーアクションで Zod などのスキーマ検証ライブラリを用い、入力値の厳密なバリデーションを行う。
- [ ] **内部エラーの隠蔽:** サーバー側のエラーメッセージ（Supabaseのエラー内容など）をクライアントにそのまま返却せず、一般的なエラーメッセージに変換する。
- [ ] **TypeScriptの厳格化:** `any` 型の使用を減らし、適切な型定義に置き換える。
- [ ] **暗黙のエラー無視の修正:** データベース操作の失敗（特にマルチステップ処理）を適切にキャッチし、ロールバックまたはエラー通知を行う。

### フェーズ 4: インフラ整備と運用体制の確立（長期）
本番環境での長期的な運用に耐えうるインフラとプロセスを整備します。

- [ ] **ベーススキーマのマイグレーション整備:** データベースを一から構築できるよう、初期状態のスキーママイグレーションファイルを作成する。
- [ ] **環境変数のドキュメント化:** `.env.example` を作成し、必要な環境変数を明確にする。
- [ ] **セキュリティヘッダーの設定:** `next.config.ts` で CSP (Content Security Policy) や HSTS などのセキュリティヘッダーを設定する。
- [ ] **バックアップ戦略の策定:** Supabaseの自動バックアップ機能の確認、および定期的なスナップショット取得など、財務データの保護戦略を策定・文書化する。
- [ ] **レートリミットの導入:** パスワードリセットやファイルアップロードなど、負荷がかかる/攻撃されやすいエンドポイントに対するレート制限を検討する。
